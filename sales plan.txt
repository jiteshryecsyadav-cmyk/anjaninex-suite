🏗 PART 1: Feature Flag Architecture (best approach)
Aap jo soch rahe ho — toggles — bilkul sahi hai. Industry me isko "Feature Flags" ya "Module Entitlements" kehte hain.
Recommended structure (2 layers):
Layer 1: Subscription Plan (template)

Predefined "Starter / Growth / Business / Enterprise" plans
Har plan ke saath default modules ON

Layer 2: Per-Firm Override (admin can customize)

Customer "Starter" plan le but admin chahe to AI on kar de extra paise me
Admin "Custom" plan bhi bana sakta hai

DB Structure:
sqlfirms table:
  - plan_code VARCHAR(20)         -- starter/growth/business/enterprise/custom
  - enabled_modules JSONB         -- { trading: true, ai: false, hr: false, ... }
  - ai_quota_monthly INT          -- 100 scans/month etc.
  - user_limit INT                -- 10 users max
  - branch_limit INT              -- 3 branches max
  - trial_ends_at TIMESTAMP
How toggles work:

Backend middleware: agar /api/hr/* hit ho aur enabled_modules.hr = false → 403 Forbidden
Frontend service: FeatureService me signals — menu/routes use karenge @if(features.has('hr'))
Shell nav: HR menu item invisible ho jayega disabled customers ke liye

What happens when toggle changes:

Admin clicks HR Module → ON → save
Backend updates firms.enabled_modules.hr = true
Customer logs in next time:

Shell nav me "HR" menu item appear ho jata hai
/hr/* routes accessible ho jate hain


Admin clicks HR Module → OFF:

Shell nav se HR menu gayab
Backend rejects /api/hr/* calls with 403
Existing data deleted nahi hota (sirf access blocked)
